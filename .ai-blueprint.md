# TrackAI System Architecture & Multi-Tenant Blueprint

## 1. System Reality & Topology
- **Frontend:** Next.js (App Router, Client-side Auth) deployed on Vercel (`https://vercel.app`).
- **Backend API:** Express Node.js Server containerized on Google Cloud Run (`https://run.app`).
- **Database & Auth:** Supabase Cloud Instance using Drizzle ORM (PostgreSQL).
- **Current Multi-Tenancy Protocol:** Custom OIDC (Auto-discovery) mapped dynamically by domain. 

## 2. Active Structural Schema (Drizzle)
- `sso_tenants`: Master lookup table mapping corporate email domains to a `supabase_provider_id` string and assigning a unique workspace master `id` (UUID).
- `items`: Sample data table. Contains an explicit `tenant_id` UUID column marked as `NOT NULL` and foreign-keyed to `sso_tenants(id)`.

## 3. Data Isolation Handshake Pipeline
1. Frontend captures user email domain, looks up `sso_tenants` on clientside, and fires `supabase.auth.signInWithOAuth`.
2. On successful handshake, Supabase passes a standard verified JWT to the browser session.
3. Every frontend data fetch attaches this bearer JWT to the Express backend header payload.
4. Express JWT Middleware decrypts the user token, extracts `req.user.email`, strips the domain, and queries `sso_tenants` to resolve the `tenantId`.
5. Express Route Layer enforces `db.select().from(items).where(eq(items.tenantId, resolvedTenantId))`.

## 4. Phase 4 Refactoring Goal (Decoupling)
- Isolate the infrastructure layer (Auth handshake, global domain extraction, and tenant resolution) away from individual route endpoints.
- Abstract the ORM query logic so that any new table added to the project automatically inherits tenant filtering without altering core platform code.
